# INDEX.md 自动更新方案审核报告

## 审核结论
**CONDITIONAL_PASS**

方案核心逻辑可行，但存在 4 个高风险漏洞、3 个中等遗漏、5 个关键设计缺陷。在落实具体改进建议前，不建议进入生产使用。

---

## 一、可行性分析

### 1.1 触发条件识别能力

| 触发条件 | 可行性 | 评价 |
|---------|--------|------|
| stage/status 变化 | ⚠️ 有条件可行 | AI 可识别自身产生的变化，但对"外部直接修改 state.json"无感知 |
| 新增 DECISIONS.md 条目 | ✅ 基本可行 | 依赖 AI 主动检查文件修改时间，无可靠通知机制 |
| 阻塞/解除阻塞 | ⚠️ 有条件可行 | 需 AI 主动判断"当前是否被阻塞"，缺乏客观标准 |
| 切出项目时兜底 | ⚠️ 依赖外部调度 | 无强制性调度约束，纯靠 AI 自觉 |

**结论**：触发条件的**识别逻辑不闭环**，依赖 AI 自我感知而非系统级事件通知。

### 1.2 更新逻辑可靠性

当前方案中，更新 INDEX.md 的动作完全由 AI 在"适当时机"执行，无以下保障：

- ❌ 无原子性保证（写入 INDEX.md 时若会话中断，会留半截内容）
- ❌ 无版本校验（无法检测并发修改覆盖）
- ❌ 无更新锁定（多 Agent 同时操作同一项目时可能冲突）
- ❌ 无回滚机制（更新错误无法恢复到上一个正确状态）

### 1.3 无法实现的部分

1. **"AI 启动时自检 nextAction 是否与 INDEX 一致"** —— 缺乏客观基准。nextAction 是自由文本，AI 无法可靠对比"一致"与否，只能做模糊匹配。
2. **"阻塞/解除阻塞"触发** —— "阻塞"是一个主观判断，AI 没有客观标准来确定当前是否处于阻塞状态。

---

## 二、遗漏分析

### 2.1 遗漏的触发条件

1. **文件缺失或损坏**：TASK.md、LOG.md、DECISIONS.md 被删除或截断时，INDEX.md 不应继续使用旧数据，但当前无触发。
2. **长时间无进展**（>24h/72h）：项目停滞但未被标记为 PAUSED，INDEX.md 仍显示"进行中"，属于静默失效。
3. **外部直接修改 state.json**：人类或外部工具直接编辑 state.json 后，INDEX.md 不会自动同步。
4. **mode 变更**：state.json 有 `mode: full/lite` 字段，影响 INDEX.md 结构，但无触发。
5. **owner 变更**：换人接手项目时，INDEX.md 的责任信息不更新。
6. **新增重要文档**（非 DECISIONS 以外的文档）：如新增 CHECKLIST.md、ARCHITECTURE.md 等，INDEX.md 的快速导航不会同步。

### 2.2 遗漏的边界情况

1. **并发操作**：两个 AI 实例同时处理同一项目（多人协作/多 Agent 场景），INDEX.md 的更新可能互相覆盖。
2. **HANDOVER 场景**：Agent 交接时（如跨会话），INDEX.md 的 `nextAction` 与交接文档的一致性无验证。
3. **递归更新风险**：更新 INDEX.md 本身的动作是否会再次触发更新？需要锁止机制。
4. **INDEX.md 自身被误删或损坏**：无恢复机制。
5. **stage 从 DONE 退回 EXECUTE**：罕见但可能发生（如甲方推翻结论），INDEX.md 无后退更新路径。

### 2.3 遗漏的风险场景

1. **state.json 与 INDEX.md 的 status/stage 字段不同步**：审核时发现 state.json 用 `"DISCUSSING"` / `"TARGET"`，但 INDEX 示例用 `"RUNNING"` / `"EXECUTE"` —— 两套字段体系不兼容，导致任何基于字段值的触发条件**全部失效**。
2. **快速导航指向不存在的文件**：DECISIONS.md 存在但 TASK.md 被重命名为 TASK-OLD.md，INDEX.md 不会自动更新链接。
3. **AI 疲劳场景**：长会话末期 token 耗尽时，INDEX.md 更新可能被跳过而非延后执行。

## 三、风险分析

### 3.1 导致 INDEX.md 没更新的情况

| 风险场景 | 发生概率 | 后果 |
|---------|---------|------|
| AI 在产生关键决策后立即被中断 | 中 | DECISIONS.md 已更新但 INDEX.md 同步失败，下次启动时 nextAction 可能指向已完成的决策 |
| AI 误判当前 stage 未变（实际已变） | 中 | 进度倒退回溯困难 |
| 触发条件定义模糊，AI 主动忽略更新 | 中高 | 体系从内部腐化 |
| state.json 被外部工具直接修改 | 低 | 但一旦发生，INDEX.md 永久失效 |
| AI 切换到其他项目后忘记切回更新 | 中 | 更新被遗漏 |

### 3.2 导致 INDEX.md 更新错误的情况

1. **字段体系不匹配**：state.json 用 `"DISCUSSING"` + `"TARGET"`，INDEX.md 用 `"RUNNING"` + `"EXECUTE"`。若 AI 以 state.json 为真相写入 INDEX，会出现无效枚举值；若以 INDEX.md 为真相，state.json 无法理解。
2. **时间戳漂移**：INDEX.md 的进展列表时间依赖 AI 自身时间感知，与 state.json 的 `updatedAt` 可能不一致。
3. **自由文本误匹配**：nextAction 是自由文本，AI 对"一致"的理解存在幻觉风险。

### 3.3 AI 会在哪些情况下"忘记"更新

1. **认为"不需要更新"**：stage 没变但实际上 nextAction 已经实现，AI 觉得是小变化不打扰。
2. **在输出结果和更新文档之间二选一**：受 token 限制，AI 可能优先返回结果而跳过文档更新。
3. **错误地认为已经更新过了**：幻觉导致跳过后续更新。
4. **进入深呼吸/重试循环后丢失上下文**：更新动作被中断且未恢复。
5. **兜底更新"切出时"定义不清**：如果 AI 在单次会话内完成所有操作且未切换项目，切出触发器永不激活。

## 四、改进建议

### 4.1 触发机制增强（高优先级）

**建议 A：引入 state.json 的 updatedAt 作为事实源**
> 每次修改 state.json 时同时更新时间戳，INDEX.md 的更新以 `state.json.updatedAt` 是否变化为唯一触发条件，而非 AI 主观判断。

**建议 B：INDEX.md 更新原子化**
```
临时写 → INDEX.md.tmp
校验内容完整性（关键字段齐全）
mv INDEX.md.tmp INDEX.md
```
避免半截写入。

**建议 C：引入更新计数器**
state.json 增加 `indexUpdateCount` 字段，每次更新 INDEX.md 后 +1。下次切入时对比计数，若未增加但 updatedAt 变了，说明上次更新失败。

### 4.2 兜底方案（高优先级）

**建议 D：AI 切入项目的强制自检流程（写入系统提示）**
每次切入项目时必须执行：
1. 读取 state.json 的 `updatedAt`
2. 读取 INDEX.md 的最后修改时间（或增加 `lastSyncedAt` 字段）
3. 若 state.json 更晚 → 强制同步 INDEX.md
4. 若不一致 → 报错而非静默覆盖

**建议 E：关键操作后强制更新，而非"适当时机"**
将 INDEX.md 更新从"可选优化"提升为"关键操作的必需副作用"：
- 写入 DECISIONS.md 后 → 必须同步 INDEX.md
- 修改 state.json.stage 后 → 必须同步 INDEX.md
两者应绑定为同一工具调用（如 `update_project_index()`），而非独立操作。

### 4.3 自检机制增强（中优先级）

**建议 F：关键路径校验**
在 INDEX.md 中增加摘要字段：
```markdown
## 元数据（勿手动修改）
- stateRef: state.json updatedAt 快照
- hash: 当前 INDEX.md 内容哈希
```
每次读取时比对 hash，不一致则报警。

**建议 G：建立 INDEX.md 最小化可读性标准**
INDEX.md 必须包含以下最小信息才视为"有效"：
- status + stage（与 state.json 一致）
- nextAction（非空）
- 最后更新时间（非空）
缺失任一则视为损坏。

### 4.4 根本性设计缺陷修复（最高优先级）

**建议 H：统一 status/stage 字段体系**
state.json 和 INDEX.md 使用完全不同的枚举值体系，这是最高风险缺陷。必须二选一：
- 方案一：以 state.json 为事实源，INDEX.md 的 status/stage 直接映射
- 方案二：废弃 INDEX.md 中的 status/stage，完全依赖 state.json，INDEX.md 只做展示

**强烈建议方案一**，但需建立明确的字段映射表：

| state.json | INDEX.md status | INDEX.md stage |
|-----------|----------------|----------------|
| DISCUSSING | IDLE | TARGET |
| RUNNING | RUNNING | PLAN |
| RUNNING | RUNNING | EXECUTE |
| PAUSED | PAUSED | （不变）|
| BLOCKED | BLOCKED | （不变）|
| DONE | DONE | DONE |

---

## 五、最终建议

### 必须修复才能上线（Blocker）

1. **统一 status/stage 字段体系**（建议 H）
2. **建立"关键操作→INDEX 同步"的强绑定**（建议 E）
3. **实现切入项目时的强制自检**（建议 D）

### 上线后持续改进（Nice to Have）

4. 引入 `indexUpdateCount` 和 `hash` 校验（建议 C、F）
5. 增加"新增文档"触发条件（建议 A 扩展）
6. 建立 INDEX.md 最小化可读性标准（建议 G）

### 总体评价

当前方案是"有基本思路但实现依赖 AI 自觉"的半成品。最核心的问题是**把 AI 的主动行为当作系统可靠性保障**，而没有建立客观的校验和强制机制。在企业级记忆体系中，这种设计会在规模化后产生大量静默失效。

建议在修复 Blocker 级问题后，用至少 5 个真实项目做 2 周压力测试，再判断是否进入生产。

# 快照压缩策略审核报告

## 审核结论
**CONDITIONAL_PASS**

方案设计思路正确，触发时机和保留策略较为合理，但**压缩算法是核心风险点**——DAG summarization 在工程上可行，但当前描述过于理想化，缺乏可靠性保障和边界处理。若能将 DAG 简化为可验证的规则集并补齐失败处理，方案可以进入生产。

---

## 可行性分析

### 1.1 触发时机 ✅ 基本可行

切出项目触发 + 30 分钟最小间隔 + 每日归档兜底，三重保障结构合理。最小间隔是防止频繁切换产生大量快照的正确设计。

唯一风险：**切出时机的检测依赖什么？** 方案未说明切出检测机制。若依赖 AI 主动感知，参考上一个 INDEX 审核报告的风险——同样存在"AI 自觉"失效的问题。建议：
- 明确切出检测是**系统级事件**（如用户主动切换、项目列表操作）而非 AI 自我感知
- 或至少在 state.json 中记录 `lastSwitchOutAt` 时间戳作为客观依据

### 1.2 保留策略 ✅ 基本可行

30 个上限配合时间衰减（7 天全量 / 8-30 天每天 1 个 / >30 天每周 1 个），逻辑自洽。软链接 `latest` 指向最新快照是好的设计，便于快速恢复。

**一个小漏洞**：最近 10 个快照"完整保留"但未定义"完整"——是完整保留压缩前的原始快照，还是保留压缩后的完整副本？结合压缩策略看，应该是后者（Level 1 是压缩后 ~5KB）。建议明确。

### 1.3 压缩算法 ⚠️ 高风险

**DAG Summarization 工程可行性：**

| 步骤 | 可行性 | 评价 |
|------|--------|------|
| 提取关键节点（决策/阻塞/状态变更/产出物） | ⚠️ 部分可行 | 决策有 DECISIONS.md 可锚定，相对可靠；其余三者依赖 AI 主观判断 |
| 构建 DAG（节点=事件，边=因果关系） | ❌ 高度复杂 | 因果关系推断是 NLP 难题，现有模型不可靠，不建议在生产路径上依赖 |
| 合并相似对话 | ❌ 无可靠实现 | "相似"缺乏客观度量标准 |
| 删除闲聊和噪音 | ❌ 无客观标准 | 噪音判断高度主观 |

**核心问题**：DAG summarization 是学术方向，但在工程实现上，当前描述是一个"调用 LLM 做摘要"的包装，本质是 prompt 工程，不是真正的 DAG。因果边、节点合并等概念在实现中极难可靠。

**Level 1/2/3 压缩率的目标**（80%）：从 50KB 压缩到 10KB，压缩率本身是可达的，但压缩结果的**质量和一致性无法保障**——不同对话内容、不同 LLM 调用参数、不同 token 预算下，压缩后的信息密度会有显著波动。企业级系统不能依赖这种不可验证的质量。

### 1.4 存储结构 ✅ 设计合理

```
snapshot/
├── metadata.json
├── latest -> 2026-03-31T07-15/   # 软链接 ✅
├── 2026-03-31T07-15/
│   ├── index.md      # 轻量快照
│   ├── summary.md    # 压缩摘要
│   └── state.json   # 状态副本 ✅
└── archive/          # 归档 ✅
```

三层分离（最新快照 / 归档）与时间衰减策略对应。state.json 副本是必要的——恢复时需要原始状态而非仅摘要。

## 完整性分析

### 2.1 80% 压缩率是否会丢失关键信息？

**会丢失，且无法证明没丢。**

| 可能丢失的信息 | 影响 |
|---------------|------|
| 试错过程（错误的思路和被否定的方案） | 高；理解为什么走这条路需要知道为什么不是那条路 |
| 推导过程（决策依据的上下文） | 高；决策的上下文往往比决策本身更重要 |
| 隐含假设 | 高；压缩后 AI 基于摘要恢复时可能补入错误的隐含假设 |
| 微小但关键的技术细节 | 中；如具体参数值、文件名、函数名，容易被"合并相似"时误删 |
| 情绪/优先级线索（"用户很不耐烦"） | 低；但在多轮对话中对 pacing 有影响 |

**最危险的场景**：在 Level 3（>50 轮，压缩 80%）中，一次长任务的压缩摘要只保留"里程碑和决策"。如果其中某个里程碑的实现依赖特定的上下文细节，而这个细节被判断为"非关键"而删除了，恢复后 AI 可能用错误的上下文重新进入，导致方向偏移。

### 2.2 metadata.json 中的 keyNodes 字段

```json
"keyNodes": ["决策：采用 CMS SOP", "进展：完成 INDEX 设计"]
```

这个字段设计很好，是压缩结果的可读锚点。但问题：
- 依赖 AI 提取，AI 疲劳或被中断时不生成
- 没有**验证机制**——如何确认 keyNodes 覆盖了所有关键点？

**建议**：keyNodes 应作为强制字段而非可选，由结构化提取保证（非自由文本摘要）。例如定义固定模板：
```json
"decisions": [{"id": "D-001", "summary": "...", "依据": "..."}],
"blockers": [{"id": "B-001", "description": "...", "resolved": true/false}],
"milestones": [{"id": "M-001", "description": "...", "completed": true/false}]
```

## 遗漏分析

### 3.1 快照失败的边界情况

| 失败场景 | 后果 | 严重度 |
|---------|------|--------|
| 写入时磁盘满 | 快照丢失，当前上下文无任何备份 | **高** |
| 写入时进程被 kill（Ctrl+C / crash） | 损坏的快照文件，可能被归档脚本误认为有效 | **高** |
| metadata.json 写入失败（但快照目录写入成功） | metadata 与实际快照不一致，恢复时找不到最新快照 | **高** |
| state.json 被锁（外部进程占用） | 快照跳过，但无告警 | **中** |
| 软链接 latest 创建失败 | 找不到最新快照，需扫描目录 | **中** |

**关键遗漏**：没有**快照写入原子性**保障。正确做法：
1. 写入临时目录 `snapshot/.tmp/2026-03-31T07-15/`
2. 写入完成后 rename 到正式路径
3. 更新 metadata.json（同样原子）
4. 更新软链接

### 3.2 压缩失败的边界情况

| 场景 | 当前方案 | 风险 |
|------|---------|------|
| LLM 调用超时 | 无描述 | 快照挂起或无限等待 |
| LLM 返回格式非法（非 JSON/非 markdown） | 无描述 | 摘要文件损坏 |
| 压缩后 sizeKB 反而变大 | 无描述 | 浪费存储且无意义 |
| keyNodes 为空数组 | 无描述 | 压缩完全失败但系统认为成功 |

**必须增加**：压缩的**降级策略**——压缩失败时，保留**完整原始快照**（不压缩），并打上 `compressionFailed: true` 标记。

### 3.3 恢复时的边界情况

| 场景 | 风险 |
|------|------|
| latest 软链接断裂 | 找不到最新快照 |
| metadata.json 中的 id 与实际目录名不一致 | 归档恢复逻辑失效 |
| 快照损坏（部分写入）但 metadata 认为有效 | AI 用错误上下文继续工作，后果不可预测 |
| 恢复时 state.json 与 snapshot/state.json 冲突 | 以哪个为准？当前未定义 |
| 用户在快照恢复后立刻切换项目 | 刚恢复的上下文还没来得及利用就被覆盖 |

### 3.4 存储管理遗漏

- **存储预警**：达到 25 个快照（~83% 上限）时无告警
- **快照删除是隐式的**：时间衰减自动删除，但无操作记录，难以审计
- **archive/ 目录清理策略未定义**：>30 天的快照进入 archive，但 archive 自身无限增长吗？

---

## 风险分析

### 4.1 快照恢复后 AI 状态不一致

**最严重的风险场景**：

```
任务 A 进行了 60 轮对话，产生完整快照（~50KB）
→ 压缩到 ~10KB，进入 Level 3
→ 用户切到任务 B
→ 一周后切回任务 A
→ 恢复 10KB 摘要
→ AI 基于 10KB 工作，可能遗漏了原始 50KB 中的某个关键上下文
→ AI 做出与一周前不一致的决策
→ DECISIONS.md 产生矛盾记录
```

这是**静默腐化**——不报错，但信息已经丢了。

**缓解方案**：引入"摘要可靠性"评分或置信度标记，让 AI 知道当前上下文的完整度。

### 4.2 DAG 压缩的幻觉风险

若 DAG summarization 最终实现为"让 LLM 总结对话"：
- 每次总结结果不同（即使同一对话）
- 关键细节被 LLM 自行"补全"
- 无法回溯验证摘要是否忠实于原始对话

**没有原始对话的完整保留**，就无法验证摘要的可信度。

### 4.3 压缩质量的不稳定性

| 对话类型 | 可压缩比例 | 风险 |
|---------|-----------|------|
| 高密度技术讨论（大量决策、参数、代码） | 低（20-30%） | 压缩丢失关键细节 |
| 低密度探索性讨论 | 高（70-80%） | 可能过度压缩 |
| 闲聊+技术混合 | 不确定 | 噪音判断主观 |

用固定压缩率目标（80%）掩盖了内容类型的差异。

### 4.4 并发操作风险

- 两个 Agent 同时处理同一项目（多 Agent 协作场景）
- 快照写入互相覆盖
- metadata.json 并发写入产生冲突

---

## 改进建议

### 5.1 压缩算法简化（最高优先级）

**用可验证的规则集替代 DAG summarization**：

```
Phase 1: 结构化提取（确定性，非 LLM）
- 从 DECISIONS.md 提取决策列表 ✅ 可靠
- 从 state.json 提取 status/stage 变更历史 ✅ 可靠
- 从 LOG.md 提取里程碑事件 ✅ 可靠

Phase 2: 启发式过滤（非严格 DAG）
- 识别"重复内容"：连续重复的消息对，删除重复
- 识别"噪音"：用户发送的纯 ACK（"好"、"OK"、"收到"）+ AI 的确认回复，成对删除

Phase 3: LLM 摘要（仅用于 Phase 1/2 之后剩余的自由文本）
- 剩余内容 < 20KB 时才调用 LLM 摘要
- 摘要结果与 Phase 1 的结构化数据合并
```

**好处**：Phase 1 是确定性的，可验证、可回溯；LLM 只处理无法结构化的部分。

### 5.2 增加快照校验（高优先级）

每次快照后立即验证：
1. 快照目录完整性（index.md + summary.md + state.json 都存在）
2. state.json 可被解析
3. snapshot sizeKB > 0
4. keyNodes 非空（压缩后）
5. 计算 snapshot 内容 hash，存入 metadata

恢复时：读取 hash 比对，不一致则告警并拒绝恢复。

### 5.3 降级策略（高优先级）

```python
def compress_snapshot(snapshot):
    try:
        result = dag_summarize(snapshot)  # LLM 调用
        if result.sizeKB > snapshot.sizeKB * 0.25:  # 压缩后反而变大
            return snapshot  # 降级：保留完整快照
        if result.keyNodes is empty:
            return snapshot  # 降级：摘要失败，保留完整
        return result
    except Exception as e:
        log_warning(f"Compression failed: {e}")
        return snapshot  # 降级：保留完整快照
```

### 5.4 增量快照替代全量快照（中优先级）

当前方案每次保存完整上下文（~50KB）。**更优方案**：
- 每个 turn 增量保存（只保存新增消息 + 当前 state）
- 快照 = 基准快照（最近的全量快照）+ 增量日志
- 恢复时重建完整上下文

**代价**：恢复时需要重放所有增量，复杂度增加；收益是存储更小、保留更完整。

### 5.5 原始对话保留（强烈建议）

**不要删除原始对话**。建议的存储策略：
```
snapshot/2026-03-31T07-15/
├── index.md         # 轻量快照（始终保留）
├── summary.md       # 压缩摘要（始终保留）
├── state.json       # 状态副本（始终保留）
└── raw.md           # 原始对话完整副本（按需保留）
```

- `raw.md` 在 Level 1/2/3 压缩时**保留原始版本**，仅在 >30 天归档时可选压缩
- 恢复时优先使用 raw.md（如果存在），否则用 summary.md
- 这样压缩失败有安全网，验证摘要可信度有基准

---

## 最终建议

### Blocker 修复（上线前必须完成）

1. **简化压缩算法**：放弃完整的 DAG summarization，改用"结构化提取 + 启发式过滤 + 可选 LLM 摘要"的降级方案。Phase 1 必须可独立运行。
2. **增加快照校验**：写入后立即验证完整性，不完整则重试或告警。
3. **压缩降级策略**：压缩失败时保留完整快照，不丢数据。
4. **原始对话保留**：引入 `raw.md`，确保摘要可验证。
5. **切出检测机制明确定义**：不能依赖 AI 自我感知切出时机。

### 重要改进（上线后持续迭代）

6. metadata.json 增加 `compressionFailed` 和 `contentHash` 字段
7. 存储达到 80% 上限（24 个快照）时告警
8. archive/ 目录自身需要定期清理（建议 >90 天移出 archive 或删除）
9. 恢复流程需要与 INDEX.md 自检机制联动（参考上一个审核报告的建议 D）
10. 考虑增量快照替代全量快照（长期架构演进方向）

### 总体评价

当前方案的核心问题与上一个 INDEX 方案类似：**把不可靠的 LLM 能力当作系统可靠性的保障**。DAG summarization 是一个好的研究方向，但不是生产系统的可靠组件。

建议的最小可行方案：
- 触发 + 保留策略：保持当前设计 ✅
- 压缩：用"结构化提取 + 固定规则"替代 LLM DAG，80% 压缩率目标改为"尽力而为，失败保留完整"
- 存储：增加 raw.md 安全网 + 快照校验

在这个基础上再谈企业级可靠性。

## 一、架构合理性分析

### 1.1 四层架构 ✅ 基本合理

| 层级 | 设计合理性 | 评价 |
|------|-----------|------|
| 项目层 | ✅ 正确 | CMS SOP 七件套是久经考验的任务封装，项目隔离是最小必要的上下文边界 |
| 会话层 | ⚠️ 设计正确，实现有风险 | snapshot/ 是对的，但压缩策略过度依赖 LLM（见快照审核报告） |

| 索引层 | ⚠️ 过于简陋 | GLOBAL-INDEX.md 只有活跃项目列表，跨项目搜索依赖人工遍历 INDEX.md，效率极低 |

### 1.2 目录结构 ⚠️ 基本清晰，有 2 处歧义

**✅ 优点**：
- 七件套命名标准化，便于工具脚本化
- snapshot/ 子目录结构清晰
- archive/ 分离归档项目

**⚠️ 问题**：

1. **INDEX.md 与 state.json 的字段体系不统一**（已在 INDEX 审核报告中标记为 Blocker）
   - state.json: `"DISCUSSING"`, `"TARGET"`
   - INDEX.md: `"IDLE"`, `"RUNNING"`, `"EXECUTE"`
   - 这导致任何基于字段值的自动化逻辑**全部失效**

2. **`HANDOVER.md` 和 `snapshot/` 的职责重叠**
   - HANDOVER.md 是交接文档，记录"换人时需要知道什么"
   - snapshot/ 是状态快照，记录"任意时刻的项目状态"
   - 两者都是为了"恢复上下文"，但格式和用途不同
   - **问题**：如果 AI 切换时只恢复了 snapshot/ 而没有参考 HANDOVER.md，可能遗漏人的因素（"为什么这么做"而非"做到哪了"）

### 1.3 复用 CMS SOP + CaS ✅ 决策正确

- CMS SOP 七件套提供了完整的任务生命周期管理，与记忆体系天然互补
- CaS（Chat Archive System）提供了会话归档的基础能力，复用而非重写是正确决策
- 唯一风险：CMS SOP 是为"单任务串行"设计的，记忆体系的核心场景是"多任务并行/切换"，两者的假设不同。当前方案未说明如何处理 SOP 的"进入/退出"与项目切换的交互。

**建议**：在 CMS SOP 的状态机中增加"PAUSED"和"RESUMED"状态（对应项目切换），使 SOP 实例能感知自己在被切换，而不是认为自己被"中断"或"完成"。

## 二、组件设计分析

### 2.1 项目切换流程 ⚠️ 有逻辑但不闭环

```
切出项目 A：
1. 检查距离上次快照是否 > 30 分钟
2. 创建快照（原子写入）
3. 更新 state.json（status=PAUSED, resume=当前进度）
4. 强制同步 INDEX.md
5. 更新 current-project.json
```

**逻辑正确，但执行依赖 AI 自觉**：

| 步骤 | 执行主体 | 问题 |
|------|---------|------|
| 步骤 1 | AI 主动检查 | 无客观触发机制，"何时检查"不明确 |
| 步骤 2 | AI 主动创建 | 无强制性调度，可能被跳过 |
| 步骤 3 | AI 主动更新 | 无锁止机制，可能被并发覆盖 |
| 步骤 4 | AI 主动同步 | 无强制约束，可能被遗忘 |
| 步骤 5 | 外部更新（current-project.json）| ✅ 这是外部行为，相对可靠 |

**关键缺失**：没有定义"谁来触发项目切换"。

- 如果是用户主动触发 → 步骤 1-4 由 AI 在收到切换指令后执行，有执行时机
- 如果是 AI 自行决定切换 → 无触发时机，可能无限拖延
- 如果是外部调度（如 cron）→ 需要外部调度系统，当前后端无此能力

**结论**：项目切换流程需要明确**触发主体**，建议以"用户主动切换"为主要场景，AI 自行切换作为次优场景（通过心跳检查触发）。

2. **触发机制依赖 AI 自觉**（Blocker）
3. **无原子性保障**（Blocker）
4. **无版本校验**（中等）

详见：上方「INDEX.md 自动更新方案审核报告」。

### 2.4 全局索引 ⚠️ 严重不足

当前 GLOBAL-INDEX.md 只包含活跃项目列表，缺少：

| 缺失功能 | 影响 |
|---------|------|
| 跨项目关键词搜索 | 无法快速定位"在哪个项目讨论过 X" |
| 项目状态聚合视图 | 无法一眼看到所有项目的整体状态 |
| 归档项目访问 | 归档项目存放在 archive/ 但无入口，需人工记忆路径 |

**跨项目搜索方案对比**：

| 方案 | 实现成本 | 可靠性 | 评价 |
|------|---------|--------|------|

**建议**：用 JSON 聚合替代纯文本 GLOBAL-INDEX.md——每个项目提供标准 JSON 输出，全局搜索用 `jq` 或简单脚本实现。

## 四、风险分析

### 4.1 导致系统失效的主要场景

| 场景 | 概率 | 后果 | 严重度 |
|------|------|------|--------|
| AI 在产生关键决策后被中断，未同步 INDEX | 高 | INDEX 与实际状态永久不一致，下次恢复从错误状态继续 | **高** |
| 快照压缩失败但系统认为成功（无校验）| 中 | 压缩结果损坏，恢复时 AI 用错误上下文工作 | **高** |
| DAG 压缩丢失关键上下文（过度压缩）| 中 | 静默腐化，AI 无法感知丢失了什么 | **高** |
| state.json 和 INDEX.md 字段体系不一致，自动化逻辑失效 | 高 | 所有基于字段值的触发/同步机制全部失效 | **高** |
| current-project.json 更新失败（外部修改 state.json）| 低 | 但一旦发生，所有后续操作针对错误项目 | **高** |
| 归档项目 >30 天被移动到 archive/，但 GLOBAL-INDEX 无记录 | 中 | 无法找回归档项目，除非人工记忆路径 | **中** |

### 4.2 静默腐化风险 ⚠️ 最高优先级

记忆体系最大的敌人是**静默腐化**——系统不报错，但信息已经丢失或错误。

**腐化路径 1**：
```
INDEX.md 写入不完整 → AI 下次读到错误的 nextAction → 基于错误上下文继续工作
→ 新决策基于旧上下文 → DECISIONS.md 产生矛盾记录 → 无法追溯哪个决策是对的
```

**腐化路径 2**：
```
快照压缩 → 关键上下文被"摘要掉" → 恢复后 AI 用错误上下文继续
→ 新产出与原始方向不一致 → RESULT.md 和实际产出不匹配
```

**腐化路径 3**：
```
快速切换（< 30 分钟）→ 不创建快照 → 直接切出 → 切回时无快照可用
→ AI 靠残余上下文工作 → 与真实状态产生漂移
```

**缓解措施**：
1. INDEX.md 写入必须原子化
2. 快照压缩必须有完整性校验
3. 30 分钟最小间隔改为"每次切出都必须快照"（间隔只是控制是否压缩）
4. 每次恢复时强制自检

### 4.3 并发和边界情况

| 场景 | 风险 | 缓解 |
|------|------|------|
| 两个 AI 实例同时处理同一项目 | INDEX.md 和快照可能被覆盖 | 引入文件锁（flock）或锁文件 |
| AI 在写 INDEX.md 时被中断 | 半截 INDEX.md | 原子写入（临时文件 + rename）|
| current-project.json 与实际不符 | 所有后续操作针对错误项目 | 每次切入时以文件系统中的 current-project.json 为准 |
| 用户快速切换多个项目（> 3 个/小时）| 产生大量快照，存储快速消耗 | 快速切换时只创建快照，不压缩 |
| AI 长会话末期 token 耗尽 | INDEX.md 更新被跳过 | 增加更新计数器，不更新则告警 |

## 五、遗漏分析

### 5.1 功能遗漏

| 遗漏功能 | 影响 |
|---------|------|
| **归档项目访问入口** | GLOBAL-INDEX 只有活跃项目，归档项目在 archive/ 但无法发现 |
| **项目搜索** | 无法通过关键词找到相关项目，只能通过 GLOBAL-INDEX 列表 |
| **经验库与项目的关联** | EXPERIENCE-POOL 是孤立的，无法知道某个经验来自哪个项目 |
| **项目模板** | 没有"新建项目"的模板，新项目初始化依赖手动创建 |
| **复盘 SOP** | 周/月复盘没有定义流程，无法自动化知识沉淀 |
| **多语言/多格式支持** | 所有文档都是中文 Markdown，无法支持英文项目或 PDF 产出 |

### 5.2 错误处理遗漏

| 遗漏 | 后果 |
|------|------|
| 快照写入失败（磁盘满）| 没有任何告警，继续执行导致无快照可用 |
| 快照写入被中断（进程 kill）| 损坏的快照文件被误认为有效 |
| 压缩后 sizeKB 反而变大 | 浪费存储且系统认为压缩成功 |
| state.json 格式非法（被外部工具破坏）| AI 无法解析，进入未知状态 |
| INDEX.md 损坏 | 无恢复机制，AI 基于错误状态继续工作 |
| current-project.json 指向不存在的项目 | 无校验，无报错，后续操作失败 |

### 5.3 用户体验问题

| 问题 | 影响 |
|------|------|
| **切入项目后需要等待 AI 输出摘要** | 如果项目很大，恢复时间长，用户等待体验差 |
| **切出时 AI 执行后台操作（快照 + 同步）阻塞用户** | 用户感知到"切换卡顿" |
| **快照压缩是隐式的，用户不知道自己的对话被"删减"了** | 透明度问题，用户可能对信息丢失不知情 |
| **没有"当前在哪个项目"的可见指示** | 用户不知道自己的指令被发到哪个项目 |

## 六、改进建议

### 6.1 Blocker 修复（上线前必须完成）

**优先级 1：统一 state.json 和 INDEX.md 的字段体系**

| state.json | INDEX.md status | INDEX.md stage |
|-----------|----------------|----------------|
| DISCUSSING | IDLE | TARGET |
| RUNNING | RUNNING | PLAN |
| RUNNING | RUNNING | EXECUTE |
| PAUSED | PAUSED | * |
| BLOCKED | BLOCKED | * |
| DONE | DONE | DONE |

**优先级 2：建立"关键操作 → INDEX 同步"的强绑定**

将 INDEX.md 同步从"AI 自觉"改为"系统强制"：
```
每次执行以下操作时，同步必须作为同一事务的一部分：
- 写入 DECISIONS.md → 同步 INDEX.md
- 修改 state.json.status/stage → 同步 INDEX.md
- 切出项目 → 全量同步 INDEX.md
```
两者应绑定为同一工具调用（如 `update_project_state()`），而非独立操作。

**优先级 3：简化快照压缩算法**

放弃 DAG summarization，改用：
```
Phase 1: 结构化提取（确定性）
- DECISIONS.md → 决策列表
- state.json → 状态变更历史
- LOG.md → 里程碑事件

Phase 2: 规则过滤
- 删除重复消息对（ACK + 确认）
- 删除纯闲聊

Phase 3: LLM 摘要（仅用于剩余自由文本，< 20KB 时）
- 失败则降级保留完整快照
```

**优先级 4：快照完整性校验**

每次快照后立即验证：
1. 所有文件存在（index.md, summary.md, state.json）
2. state.json 可解析
3. sizeKB > 0
4. 压缩后 keyNodes 非空

**优先级 5：增加 raw.md 安全网**

原始对话完整保留，按需使用而非强制压缩。

---

### 6.2 中等优先级改进

**优先级 6：明确定义项目切换的触发主体**

建议：以用户主动切换为主要场景。当用户发出切换指令时：
1. AI 收到指令，立即执行切出流程（快照 + 同步 + 更新 state.json）
2. AI 清空上下文
3. 读取新项目的 state.json + INDEX.md
4. 输出项目进展摘要

AI 自行切换作为次优场景，通过心跳检查（距离上次快照 > 30 分钟 + 超过 5 轮无相关操作）触发。

**优先级 7：升级 GLOBAL-INDEX.md**

用 JSON 聚合替代纯文本列表：
```
projects/
├── index.json          # 所有项目元数据的聚合索引
│   └── GLOBAL-INDEX.md # 人类可读展示（从 index.json 生成）
```

`index.json` 结构：
```json
[{
  "id": "SOP-20260330-001",
  "name": "企业级Agent记忆体系",
  "status": "RUNNING",
  "lastUpdated": "2026-03-31T07:15:00Z",
  "nextAction": "完成索引层设计",
  "archive": false
}]
```

`jq 'map(select(.status == "RUNNING")) | sort_by(.lastUpdated) | reverse'` 即可实现活跃项目排序搜索。

**优先级 8：增加错误处理和告警**

| 场景 | 处理 |
|------|------|
| 快照写入失败 | 告警 + 拒绝切出 |
| state.json 格式错误 | 拒绝切入，提示修复 |
| 存储达到 80% 上限 | 告警 |
| 快照压缩失败 | 保留完整快照 + compressionFailed: true |
| INDEX.md 校验失败 | 强制从 state.json 重建 |

**优先级 9：归档项目可发现性**

GLOBAL-INDEX.md 增加"归档项目"部分，或提供 `archive/index.json` 供查询。

### 6.3 长期演进建议

**优先级 10：知识层设计**

周/月复盘的 SOP + 模板，EXPERIENCE-POOL 的沉淀流程。参考：
- 周复盘：从本周的 LOG.md + DECISIONS.md 提取进展和决策
- 月复盘：从本月的周复盘 + 项目 INDEX.md 提取主题和教训
- 经验池：跨项目通用经验提取（如"如何处理多 Agent 协作中的上下文冲突"）

**优先级 11：外部调度能力**

每日归档（23:00）需要 cron 或类似机制。当前 OpenClaw 的 cron 支持是否可用？如果可用，建议：
- 每日 23:00 扫描所有 PAUSED 项目，创建或更新快照
- 每月最后一天运行归档脚本，将 >30 天无更新的项目移入 archive/

**优先级 12：考虑增量快照**

当前每次保存完整上下文（~50KB），存储随对话线性增长。长期建议：
- 基准快照（全量）+ 增量日志（每个 turn 的新增内容）
- 恢复时重放增量
- 收益：存储更小，保留更完整

## 总体评价

**当前方案的核心问题**：把 AI 的主动行为当作系统可靠性的保障，而不是建立客观的校验和强制机制。

在单用户、单 Agent 的理想情况下，当前设计可以工作。但在企业级的多任务、长时间、高价值工作场景下，"AI 自觉"必然产生大量静默失效。

**最小可行方案**（满足 Blocker 修复后）：
- 项目层：CMS SOP 七件套 ✅（已可用）
- 会话层：快照 + 简化压缩 + raw.md 安全网 ✅（修复后可用）
- 知识层：暂时搁置，先跑通项目层 + 会话层
- 索引层：升级 GLOBAL-INDEX.md + JSON 聚合 ✅（修复后可用）

建议先实现最小可行方案，用至少 5 个真实项目跑 2 周，再评估知识层的投入产出比。
