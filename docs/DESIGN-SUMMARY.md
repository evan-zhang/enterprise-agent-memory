## 一、问题定义

### 真实场景
- 员工工作不是串行的，是多任务交替推进
- 任务 A 做到一半需要暂存，切换到任务 B
- 任务 B 做到一半可能又要切回任务 A
- 切换时不希望上下文混乱或丢失

### 核心需求
1. **多任务交替推进不丢失上下文**
2. **任务级信息流可快速定位和切换**
3. **跨任务经验可沉淀复用**

### 类比人的记忆
- 统一存储（所有记忆在一个大脑里）
- 按需提取（能从海量记忆中快速提取某个项目的内容）
- 经验沉淀（跨项目的共有知识能提炼、强化、复用）

## 二、架构方案

### 核心理念
- **目录即项目**：每个项目一个独立目录
- **索引即记忆**：项目索引文件 = AI 的项目级记忆
- **切换即加载**：切换项目 = 切换工作目录 + new session + load 索引

### 四层架构

| 层级 | 职责 | 实现 |
|-----|-----|-----|
| **项目层** | 项目隔离、状态管理 | CMS SOP 七件套 + state.json |
| **会话层** | 实时归档、快照恢复 | CaS + snapshot/ |
| **知识层** | 经验沉淀、跨项目共享 | 周复盘、月复盘、EXPERIENCE-POOL.md |
| **索引层** | 跨项目搜索、项目发现 | GLOBAL-INDEX.md |

### 复用现有能力
- **CMS SOP**：项目级结构化存储（七件套）
- **CaS**：会话级实时归档（日志 + 复盘）

## 三、核心组件

### 3.1 INDEX.md 自动更新机制（v3）

**问题**：INDEX.md 失效 = AI 无法正确恢复项目上下文 = 整个体系崩溃

**解决方案**：

| 机制 | 说明 |
|-----|-----|
| **字段映射** | 以 state.json 为事实源，INDEX.md 只做展示映射 |
| **强绑定** | 关键操作（修改状态、新增决策）必须同步 INDEX |
| **切入自检** | 每次切入时比对 state.json.updatedAt 和 INDEX.md.stateRef |
| **原子性保护** | 临时写 → 校验 → mv，避免半截写入 |
| **更新计数器** | updateCount 用于检测同步失败 |

**字段映射表**：

| state.json | INDEX.md status | INDEX.md stage |
|-----------|----------------|----------------|
| DISCUSSING | IDLE | TARGET |
| RUNNING | RUNNING | PLAN/EXECUTE |
| PAUSED | PAUSED | * |
| BLOCKED | BLOCKED | * |
| DONE | DONE | DONE |

### 3.2 快照压缩策略（v2）

**问题**：完整对话（~50KB）太大，需要压缩到 ~10KB

**解决方案**：

| 机制 | 说明 |
|-----|-----|
| **三阶段压缩** | 结构化提取（确定性）+ 启发式过滤 + 可选 LLM |
| **raw.md 安全网** | 保留原始对话，确保摘要可验证 |
| **降级策略** | 压缩失败时保留完整快照 |
| **原子性保护** | 临时目录写入 → 校验 → rename |
| **完整性校验** | contentHash + compressionFailed 标记 |

**保留策略**：
- 最近 10 个快照：完整保留
- 时间衰减：7 天全量，8-30 天每天 1 个，>30 天每周 1 个
- 最大 30 个快照

### 3.3 全局索引（v1）

**问题**：项目多了如何快速找到相关项目？

**解决方案**：
- 放弃 RAG（过度设计）
- GLOBAL-INDEX.md 只列出活跃项目
- 跨项目搜索：遍历活跃项目的 INDEX.md，关键词匹配
- 历史项目：归档到 archive/，需要时手动指定

## 四、项目切换流程

### 切出项目 A
1. 用户主动说"切换到 XXX"
2. AI 检查距离上次快照是否 > 30 分钟
3. 创建快照（原子写入）
4. 更新 state.json（status=PAUSED, resume=当前进度）
5. 强制同步 INDEX.md
6. 更新 current-project.json

### 切入项目 B
1. /new 清空上下文
2. 读取 current-project.json
3. 读取 B/state.json + B/INDEX.md
4. 自检：state.json.updatedAt > INDEX.md.stateRef？→ 强制同步
5. 检查 snapshot/metadata.json 的 compressionFailed？→ 加载 raw.md
6. 输出项目进展摘要
7. 等待用户指令

## 六、核心设计原则

1. **AI 为主，用户为辅** — 自动更新，自动保存，自动提炼
2. **强绑定强制机制** — 关键操作必须触发 INDEX 同步
3. **原子性保障** — 临时写 → 校验 → 原子替换
4. **降级策略** — 压缩失败保留完整快照
5. **简化优先** — 放弃复杂方案（DAG/RAG）

### 核心判断
> 当前方案把 AI 的主动行为当作锦上添花，但系统可靠性依赖客观强制机制。
