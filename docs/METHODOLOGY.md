# 项目方法论

本文档说明我们在「企业级 Agent 记忆体系」项目中使用的方法论。

---

## 核心原则

### 1. 追本质、看本体、要最小

- **追本质**：先搞清楚问题的本质是什么，不要被表象迷惑
- **看本体**：用本体论分析问题的要素和关系
- **第一性原理**：从基本假设出发，而不是类比或模仿
- **最小原型**：先跑通最小可行方案，再迭代优化

### 2. 敢泼冷水，不迎合

- 异想天开时主动拉回来
- 追求真正有用，不追求让用户满意
- 说真话，即使不舒服

### 3. 说过就做，不放空炮

- 承诺的事跟进到底
- 不确定的先说不确定
- 做不到的先说做不到

## 项目流程（CMS SOP 六阶段）

我们使用 CMS SOP（标准化操作流程）管理项目生命周期：

```
TARGET → PLAN → CHECKLIST → EXECUTE → ARCHIVE → DONE
```

### 阶段说明

| 阶段 | 说明 | 产出 |
|-----|-----|-----|
| **TARGET** | 目标定义 | 明确要解决什么问题、成功标准是什么 |
| **PLAN** | 计划制定 | 架构设计、组件设计、实现路径 |
| **CHECKLIST** | 确认单 | 是否要做、谁来做、什么时候做、范围是什么 |
| **EXECUTE** | 执行实施 | 写代码、测试、部署 |
| **ARCHIVE** | 归档 | 整理文档、总结经验、更新知识库 |
| **DONE** | 完成 | 关闭项目、移交维护 |

### 当前状态

本项目已完成 **TARGET** 和 **PLAN** 阶段：

- ✅ TARGET：问题定义、核心需求、成功标准
- ✅ PLAN：架构设计、三个核心组件设计、审核改进
- ⏳ CHECKLIST：待确认（是否要做、实现范围）
- ⏳ EXECUTE：待实现

## 审核机制

### 为什么需要审核？

AI 生成的设计方案可能存在：
- 遗漏的边界情况
- 过度复杂的设计
- 依赖 AI 自觉的不可靠机制
- 静默腐化风险

### 审核流程

```
1. 主 Agent 完成设计 → sessions_spawn 子 Agent 审核
2. 子 Agent 独立审核 → 输出审核报告
3. 主 Agent 根据审核报告改进设计
4. 重复直到通过（PASS 或 CONDITIONAL_PASS）
```

### 审核维度

1. **可行性**：技术上是否可实现？
2. **完整性**：有没有遗漏？
3. **风险**：有没有可能导致系统失效的场景？
4. **改进建议**：有没有更简单可靠的方案？

### 审核结论

- **PASS**：可以直接执行
- **CONDITIONAL_PASS**：修复 Blocker 后可执行
- **FAIL**：需要重新设计

## 决策记录

每个重要决策都要记录：

| 时间 | 决策内容 | 备选方案 | 选择理由 |
|------|----------|----------|----------|
| ... | ... | ... | ... |

**目的**：
- 回溯时知道为什么这么做
- 新成员能快速理解决策背景
- 避免重复讨论已决策的事项

## 协作规范

### 对于 AI Agent 协作者

1. **阅读文档**：先读 `docs/DESIGN-SUMMARY.md` 了解整体设计
2. **遵循原则**：追本质、看本体、要最小
3. **敢于质疑**：发现问题要主动指出
4. **记录决策**：每个重要决策都要更新决策表
5. **审核后再执行**：任何实现前先完成审核

### 对于人类协作者

1. **Review 审核报告**：关注 Blocker 问题
2. **确认 CHECKLIST**：明确要不要做、做什么
3. **验证最小原型**：先跑通再投入大规模开发
4. **更新知识库**：完成后更新 Obsidian 和 GitHub
